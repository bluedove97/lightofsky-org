# CLAUDE.md — 홈페이지 제작 마스터 가이드

> **이 파일의 목적:** "홈페이지 만들어줘"라고 하면 처음부터 끝까지 순서대로 진행한다.
> 
> Claude Code가 전체 프로세스를 관리하고, 사용자가 직접 해야 할 부분은 상세히 안내한다.

---

## Workflow (`workflows/`)

쉬운 말로 쓰여진 단계별 지시서. 목표가 뭔지, 어떤 입력값이 필요한지, 어떤 Tool을 돌릴지, 결과물이 어때야 하는지, 문제가 생기면 어떻게 하는지. 

첫 출근날 받는 업무 매뉴얼이라고 생각해. 한 번이라도 처리한 업무는 Workflow가 생긴다.

모든 지시와 이에 대해 처리한 결과를 markdown file로 기입하고 workflow폴더에 저장하고 나중에 필요하면 참고한다.


## 🚀 시작하기

사용자가 홈페이지 제작을 요청하면, 아래 **5개 Phase**를 순서대로 진행한다.

```
Phase 1: 기획 (인터뷰)
    ↓
Phase 2: 프론트엔드 개발
    ↓
Phase 3: 백엔드 연동
    ↓
Phase 4: 관리자 페이지 (선택)
```

**각 Phase 완료 후 사용자에게 확인받고 다음 Phase로 진행한다.**

---

# Phase 1: 기획 (인터뷰)

## Claude Code가 할 일

1. **AskUserQuestionTool**로 인터뷰 진행
2. 사이트 유형 파악 후 맞춤 질문

## 첫 번째 질문: 사이트 유형 파악

```
어떤 유형의 웹사이트를 만드시나요?
답변: 교회 홈페이지 웹사이트

```

---

### 브랜드 로고 (선택)

| 질문 | 안내 |
|------|------|
| 브랜드 로고가 있으신가요? | 있으면 아래 안내대로 폴더에 넣어주세요 |

```
📋 브랜드 로고 등록 가이드

1. 프로젝트 폴더에 brand_logo 폴더를 만드세요
   
   📁 프로젝트폴더/
   └── 📁 brand_logo/
       └── logo.png (또는 .svg, .jpg)

2. 로고 파일을 brand_logo 폴더에 넣어주세요

3. 권장 사항:
   - 형식: PNG (투명 배경) 또는 SVG
   - 크기: 가로 200~500px
   - 파일명: logo.png, logo.svg 등

4. 완료되면 알려주세요!
   → 랜딩페이지 헤더에 자동으로 적용됩니다
```

---

## 교회 웹사이트 인터뷰

| 영역 | 확인 사항 |
|------|----------|
| **1. 타겟 & 목적** | 누구를 위한, 어떤 목적의 사이트인지 |
| **2. 필요한 페이지** | 메인, 교회소개, 예배안내, 게시판, 앨범 등 |
| **3. 참고할만한 사이트** | 기존 사이트 혹은 참고할 만한 사이트 |
| **4. 핵심 기능** | 교회소개, 예배안내, 게시판 등 |
| **5. 디자인 방향** | 참조 이미지, 색상, 톤앤매너 |
| **6. 백엔드 필요 여부** | 게시판, 갤러리 등 |
| **7. 관리자 페이지 필요 여부** | 게시판 관리, 갤러리 관리 등 |

---

## 인터뷰 규칙 (공통)

- **AskUserQuestionTool** 사용
- 한 번에 **최대 2~3개 질문**
- 이미 정한 내용을 **다시 묻지 말 것**
- "모르겠다" 또는 "알아서 해줘" → **합리적 기본값 제안 후 확인**
- **인터뷰 종료** 시점: 모든 필수 영역 완료 또는 사용자가 "인터뷰 종료"

**→ 사용자 확인 후 Phase 2로 진행**

# Phase 2: 프론트엔드 개발

## Claude Code가 할 일

1. `brand_logo/` 폴더 확인 → 로고 있으면 헤더에 적용
2. HTML/CSS 코드 작성
3. 반응형 디자인 적용
4. 버튼 링크 연결 (신청하기 → 결제 페이지)
5. 스크린샷 비교 검증 (최소 2회)

### 로고 적용

```html
<!-- brand_logo 폴더에 로고가 있으면 자동 적용 -->
<header>
  <img src="./brand_logo/logo.png" alt="브랜드 로고" class="h-10">
  <!-- ... -->
</header>
```

## 참조 이미지 규칙

- **참조 있음**: 정확히 일치시킬 것. 개선/추가 금지.
- **참조 없음**: 아래 가드레일 따라 고퀄리티 디자인

## 안티-제너릭 가드레일

| 항목 | ❌ 금지 | ✅ 해야 할 것 |
|------|--------|--------------|
| **색상** | Tailwind 기본색 (indigo-500, blue-600) | 커스텀 브랜드 색상 |
| **그림자** | `shadow-md` 단일 사용 | 색상 틴트된 레이어드 그림자 |
| **폰트** | 제목/본문 같은 폰트 | 디스플레이 + 산세리프 조합 |
| **애니메이션** | `transition-all` | `transform`, `opacity`만 |
| **인터랙션** | hover 상태 없음 | hover, focus, active 전부 |

## 버튼 상태 디자인 (필수)

모든 버튼에 아래 상태를 구현할 것:

```css
/* 기본 상태 */
.btn { background: #브랜드색상; }

/* 마우스 올림 */
.btn:hover { background: #진한색상; transform: scale(1.02); }

/* 포커스 (키보드 접근성) */
.btn:focus { outline: none; ring: 2px; ring-color: #브랜드색상; }

/* 클릭 중 */
.btn:active { transform: scale(0.98); }

/* 비활성화 */
.btn:disabled { opacity: 0.5; cursor: not-allowed; }
```

## 버튼 링크 연결

```html
<!-- 신청하기 버튼 → 결제 페이지로 이동 -->
<a href="https://결제페이지URL.com" target="_blank">
  <button class="btn-primary">지금 바로 신청하기</button>
</a>

<!-- 무료 상담 버튼 → 페이지 내 폼으로 스크롤 -->
<a href="#consultation-form">
  <button class="btn-secondary">무료 상담 신청</button>
</a>
```


## 출력 기본값

- 단일 `index.html`, 스타일 인라인
- Tailwind CSS CDN: `<script src="https://cdn.tailwindcss.com"></script>`
- 플레이스홀더 이미지: `https://placehold.co/너비x높이`
- 모바일 퍼스트 반응형

## 스크린샷 검증

```
1. localhost에서 서빙 (node serve.mjs)
2. 스크린샷 촬영 (node screenshot.mjs http://localhost:3000)
3. 참조와 비교 → 불일치 수정
4. 최소 2회 반복
```

## Phase 2 완료 체크

```
□ 모든 페이지 HTML/CSS 완성
□ 반응형 디자인 적용
□ 버튼 상태 (hover, focus, active, disabled) 구현
□ 스크린샷 비교 검증 완료 (최소 2회)
```

**→ 사용자 확인 후 Phase 3로 진행**

# Phase 3: 백엔드 개발

> **백엔드가 필요 없으면 Phase 4로 건너뛴다.**

---

## 왜 백엔드가 필요한가?

```
[백엔드 없이]
게시판, 앨범 조회 → ❌ 아무것도 없거나 고정된 html만 나온다

[백엔드 있으면]
관리자가 게시글 작성 → 제출 → ✅ 데이터 저장 → ✅ 사용자 페이지에서 확인 가능
```

---
### Claude Code가 할 일

1. 백엔드는 python으로 작성
2. 데이터베이스는 MYSQL 사용. 이미 설치된 데이터베이스 접속정보가 있음
3. 테이블 생성 SQL 제공
4. 폼 → DB 저장 연동
5. 관리자 페이지에서 데이터 조회 연동

### 👤 사용자가 직접 할 일
1. 클로드가 알려준 테이블 생성 스크립트를 DB에 적용한다.



### ⚠️ 보안 규칙

- ❌ API 키를 코드에 직접 입력 금지
- ❌ .env 파일을 GitHub에 커밋 금지
- ✅ .gitignore에 .env 추가 필수

## Phase 3 완료 체크

```
□ 테이블 생성 SQL 실행 완료
□ 게시글 제출 → DB 저장 테스트 성공
□ 사용자 페이지에서 게시글 내역 확인 가능
□ .env.local 생성, .gitignore 확인
```

**→ 사용자 확인 후 Phase 4로 진행 (필요 시)**
